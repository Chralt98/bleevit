# Design Review: BACKEND-PLAN.md + FRONTEND-PLAN.md as one system

## Summary verdict

These are two unusually rigorous documents that do not yet describe one system. The factual core they share is solid — I cross-checked every constraint the frontend extracts from the backend (bounds, storage shapes, state machines, decimals, crank names, the finality note) and found essentially no misquotes. But the system falls apart at three seams: **(1)** the entire integration contract — the `FutarchyApi` runtime API, the `RecentCohortSummaries` ring, the `Traded`/`Observed` market events, and the frontend's §30 patch — exists only in the frontend document; the backend as written contains none of it (0 of 11 patch items applied). **(2)** The backend's solvency-critical core has two design-level correctness holes: its conditional-ledger type system cannot represent the gate-market instruments and Baseline market its own decision engine requires. **(3)** Neither document covers the values-governance layer's user experience — the backend makes referenda mandatory for upgrades and oracle adjudication, and the frontend builds zero referenda/voting functionality.

Verdict: **not implementation-ready as a pair.** The backend needs a ledger rework and the two teams need a contract freeze before either §26 work breakdown starts. The good news is that both docs are honest about their unknowns ([VERIFY] discipline, gated prototypes), and most fixes are specification work, not architectural rewrites.

## Cross-cutting issues (where the two plans disagree or leave a hole between them)

**X-1 (Critical) — The integration contract is one-sided.** The frontend requires eleven `FutarchyApi` methods (FRONTEND-PLAN.md §13.1); the backend's only commitment is a directory comment — `runtime-api/ # sp_api declarations (market quotes, epoch status, NAV, welfare)` (BACKEND-PLAN.md:1213) — which covers 4 of the 11. `proposal_summaries`, `decision_stats`, `account_positions`, `execution_queue`, `params`, `recent_cohorts`, and `open_oracle_rounds` are absent, as are the `RecentCohortSummaries` storage item and the `Traded{market, side, amount, cost, p_after}` / `Observed{market, o_t}` events. The events are the sharpest edge: the frontend's entire price-history pillar (local index, charts, the E6/E7 UX rows) is built from those two events, and without them it collapses to the 8 TWAP checkpoints — the frontend says so itself in §15.3. FE-R1 is the root of the frontend's work breakdown, and there is no contingency if the backend team rejects §30.

**X-2 (Critical) — The values layer has no frontend.** The backend mandates a values `ratify` referendum for every CODE/META upgrade (§18.3), makes terminal oracle adjudication an `OracleResolution` referendum (§15.2), and elects/recalls guardians via referenda (§16, §20). The frontend contains zero occurrences of referenda, conviction voting, or ratification — no screens, no extrinsics, nothing — while claiming to serve "every protocol workflow" (INV-FE-4). Either half the governance model runs on polkadot-js Apps, or the frontend scope is wrong. This compounds a backend-internal problem (B-8 below): the ratification timing is arguably infeasible as specified, so nobody has yet designed the flow the missing UI would implement.

**X-3 (High) — Nobody actually owns deep history.** The backend says `ExecutionRecords` are "pruned to indexer" (BACKEND-PLAN.md:366) — the indexer is the sole retention layer, despite being labelled "never consensus-relevant". The frontend's patch P-4 rewords this to "event-derived history reconstructible by any client" — but the frontend's own catch-up/backfill design (§15.4, §15.6) assumes a light client can read events at arbitrary historical depth, which fails against default state pruning (~256 blocks ≈ 25 minutes). Frontend §19 E3 claims a 2-hour gap is closed by "local-index catch-up" — already past default pruning. So the backend delegates history to an indexer the frontend declares optional, and the frontend's replacement mechanism can't reach the past either. U-3's claim that archive nodes are "not a frontend dependency" is false for any gap deeper than peer retention.

**X-4 (High) — The browser can't reach the chain as provisioned.** The frontend requires ≥8 browser-reachable WSS(443)/WebRTC bootnodes run by ≥4 operators (§11.3, §28). The backend's node-roles table (§4.2) has no bootnode row, no WSS anything — and the frontend's own §30 patch, which edits exactly that table, doesn't add one. Without coordination, the canonical frontend cannot connect at all.

**X-5 (High) — Oracle events don't exist.** The frontend ingests "oracle report/challenge/adjudication" events (§15.3) for dispute history; the backend's oracle pallet (§5.2.5, §15) defines calls but not a single event name — and unlike `Traded`/`Observed`, no §30 patch item adds them.

**X-6 (Medium) — Forecast-trading contradiction.** The backend explicitly keeps live-branch books trading through measurement "as a running forecast" (§11.1, §11.5). The frontend's pre-sign precondition table (§17.6) only permits `market.buy/sell` when the proposal is in `Trading/Extended` — so the frontend would refuse to construct transactions the chain considers valid, silently disabling a designed feature.

**X-7 (Medium) — Emergency upgrades brick the canonical frontend mid-epoch.** The backend has an EMERGENCY proposal class and guardian playbooks that can move fast. The frontend's release train requires a 72-hour staging soak, ≥2 independent build attestations, and a 3-of-5 ArNS repoint (§24.4), and it disables signing globally when descriptors don't match the runtime (`ReadOnlyIncompatible`). An expedited runtime upgrade during a live 21-day epoch leaves users with funds in open markets and time-boxed obligations (oracle challenge windows, redemptions) unable to sign for potentially days. No expedited-release procedure exists.

**X-8 (Medium/Low, batch) — Smaller contract drift.** The Baseline market's `MarketId` is undiscoverable — frontend reads it "from `EpochOf`/epoch storage" but the backend's `EpochInfo { index, phase, phase_start_block }` has no such field (this connects to backend B-2 below: Baseline is under-specified on both sides). The backend never states an ss58 prefix; the frontend pins one as a build constant. The frontend reads `Assets.Account` but the backend's pallet-assets instance is named `ForeignAssets` — a typed-API key that won't exist under the frontend's spelling. Frontend §15.3 uses event shorthand (`Withdrawn`, `Cancelled`…) where the backend emits `ProposalWithdrawn`, `ProposalCancelled`…. The frontend hardcodes the 64-position bound in views and preconditions while the backend makes it a tunable default. The frontend has no NUM onboarding flow (XCM reserve transfer from Asset Hub) and never states funding is out of scope. Both docs cite a backend §18.6 that doesn't exist (§18 ends at 18.4).

## BACKEND-PLAN.md — risks and open questions, by severity

**Critical**

- **B-1 — Gate-market instruments are unrepresentable in the conditional ledger.** §13 requires YES/NO gate complete sets per branch for CODE/META/high-TREASURY proposals, but §7 fixes `PositionKind ∈ {BranchNum, Long, Short}` (6 kinds/vault), `VaultInfo.scalar_sets: [Balance;2]` holds one scalar dimension per branch, and the §10.2 conservation identity `E = B + Q_Accept + Q_Reject` has no gate term. The pallet the spec calls "sole custodian of market collateral" cannot custody the instruments the decision engine needs. This invalidates the I-1 identity as stated.
- **B-2 — The Baseline market has no collateral model.** §11.1 mandates an unconditional per-epoch Baseline market and §14 makes its TWAP load-bearing (`r_eff = max(r_f, base − σ)`), but `PositionId` requires a `branch` and vaults are keyed by `ProposalId` — an unconditional market fits neither. What collateralizes Baseline LONG/SHORT and how it settles is unspecified.

**High**

- **B-3 — Normative LMSR vector V1 is wrong.** §11.6 claims `10000·ln((e^{0.1}+1)/2) = 512.925464970…`; the true value is **512.4948…** (V2–V5 check out). Since implementations "MUST reproduce to stated precision" and these vectors seed the differential-test corpus — including the frontend's TypeScript port (§14.5) — a wrong constant either fails conformance or propagates a wrong implementation to both sides.
- **B-4 — No VOID vault state.** §10.5's VOID semantics (both branch-NUM kinds redeem 1:1; both branches' scalar sets at s=0.5) are representable by none of `{Open, Resolved(Branch), ScalarSettled}`, and no calls/errors exist for VOID redemption. This also propagates into the frontend, whose precondition tables hardcode the three-state vault model.
- **B-5 — Gate-breach determinism contradiction.** ADR-10 claims breach facts are "deterministic on-chain… no oracle discretion," but the C pillar (§12.3) includes attested incident and external-price components with multi-day challenge windows. Either daily flags use a different C than welfare-C (two definitions of one gate), or gate settlement inherits oracle discretion. Ambiguous and load-bearing.
- **B-6 — GOV reflexivity leaks in via the E component.** The kernel excludes protocol-market and GOV prices from W (§12.2), but `E` values GOV-denominated collator/guardian bonds using an attested GOV price, which flows into C → gates → binding decisions — the exact reflexive lever the exclusion exists to remove.
- **B-7 — Trading denomination is unstated, and annulment depends on it.** G-3/I-2 (annulment) only hold if trades are denominated in branch-NUM (trader splits NUM, holds the mirror). §11.3 never says whether `buy()` takes NUM or branch-NUM, or whether buyers receive the mirror. If buyers pay plain NUM for conditional LONG, the "economically reverted" guarantee fails for market buyers. The related "cannot be insolvent" claim (§11.3) also under-specifies the continuous revenue-splitting mechanic it depends on, and `headroom` is never sized.
- **B-8 — Ratification timing is circular.** §18.3(2) requires the ratify referendum "Passed by maturity (checked at queue time…)", but queue time is decision close and the track needs ≥9 days — so values must ratify an adoption before markets have adopted it, against an unspecified preimage, submitted by an unspecified party. As written, the queue-time check can never pass.
- **B-9 — `Positions` is not globally bounded.** Per-account it is; globally it scales with account count, and splits are refundable, so an attacker pays only fees + ED per account to bloat state. Reaping at `ReapBatch = 100` after a 1-year delay can't keep up. TM-12's "every map bounded; bonds price entries" is false for account-keyed maps.
- **B-10 — Dispute-griefing veto.** §14.1 step 2 rejects (ProcessHold) any proposal whose `MetricSpec` is touched by *any open dispute* at its decide block. One challenge bond, timed to a decide window, censors every proposal on that spec for an epoch — repeatable, status-quo-safe, and absent from the threat model.
- **B-11 — The entrenchment attestation is presence-only.** §18.4 is honest that only the attestation's *presence* is checked and content verification is social — but there's no attestor registry, bond, or challenge, so any proposer attaches an arbitrary hash. I-19's table row oversells this as an enforced invariant.
- **B-12 — Phase-3 insider risk is unmodeled.** Phase 3 runs real mainnet NUM markets while `pallet-sudo` (4-of-6 founding multisig) is live and call filters bind only "post-bootstrap." TM-1…26 has no malicious-insider/multisig-compromise row for that window.
- **B-13 — Untreated bootstrap economics.** GOV genesis distribution (amounts, concentration, vesting) is unspecified even though every values-layer guarantee depends on it; treasury bootstrap sizing is absent even though worst-case LMSR subsidy across concurrent books (~800k NUM/epoch at defaults for 5 META proposals) implies NAV ≥ ~100M for the 0.75% cap — "slots shrink to fit" means small treasuries run ~0 proposals.

**Medium (selected)**

- Slot-crowding DoS via fully-refundable bonds: 5 max-bond junk proposals per epoch crowd all slots, fail decision-grade, refund, repeat — pure opportunity-cost attack that also drains POL (§8.1 T10/T5).
- Keeper economics never close arithmetically: required observation coverage across up to ~121 books vs a 3,000 NUM/epoch budget, with no batched crank (§4.2, §11.8, §21).
- Three mutually inconsistent live-market bounds (31/epoch → "≤121"; `MaxLiveMarkets = 225`; `MaxMarketsPerProposal = 7` with "margin" undefined) and inconsistent cohort decompositions (§11.1, §5.2.2, §9.2, ADR-4); `IntakeQueue = 64` vs `MaxLiveProposals = 32` "(all states)".
- USDC issuer/Asset Hub freeze of the sovereign account strands 100% of NUM backing; PB-DEPEG doesn't cover it (§17.1, TM-8).
- Preimage TOCTOU: nothing stops a proposer `unnote`-ing after adoption to renege post-pricing (§18.2).
- On-demand coretime fallback distorts every block-denominated window while the dead-man switch counts relay blocks — spurious triggers possible (§4.1, §12.5).
- Enum/state drift: `SecondExtensionFailed` never returned, `PayloadReverted`/`VetoUpheldByReview` unreachable, `decide_at` not a `Proposal` field, `DispatchOutcomeCode` undefined (§14, §7, §8.1); `epoch.length` is META-mutable with no in-flight-cohort rule.

**Open questions:** 14 [VERIFY] tags (Paseo onboarding, coretime renewal mechanics, pallet-assets holds on 2603, USDC asset ID/location, migration failure semantics, HRMP setup, etc.); invariants I-22 and I-24 are enforced only by review/CI convention, not runtime mechanism.

## FRONTEND-PLAN.md — risks and open questions, by severity

**High**

- **F-1 — Historical reads at depth are unhandled.** Catch-up and backfill assume `eventsAt(hash)` works at arbitrary depth through a light client; default peers prune state after ~256 blocks. §16.4 acknowledges depth limits, but §15.4 has no "gap too deep" state and §19 E3 promises catch-up over a gap that's already unreachable. Consequences: silent permanent holes for returning users, the 50 blocks/s backfill premise, and the false U-3 claim (see X-3).
- **F-2 — The RPC-fallback promotion rule is a verification hole.** §11.8 promotes RPC-served data to `verified-finalized` if smoldot "cross-checks the finalized hash at the same height" — but a matching block hash authenticates the header, not RPC-served storage values. As written, unverified state can enter the exact trust tier the transaction path relies on. (Probably intended as "re-fetch via smoldot," which should be stated — and then the RPC contributed nothing to verified data.)
- **F-3 — The spec's normative backbone is unpublished.** INV-FE-1…15 are cited 79 times and defined nowhere in the repo; likewise the "required question blocks," diagram/skeleton numbering, and decision biases. §32's compliance table verifies against invariants a reader cannot read. The document is not independently auditable.
- **F-4 — Hard backend dependency with no fallback.** `Traded`/`Observed` events and `RecentCohortSummaries` are hard requirements for price history and first-visit settlement dashboards (§13, §15.3); FE-R1 gates the whole WBS; P-10 additionally imposes a standing release veto on the backend train. No contingency is specified if §30 is rejected (see X-1).
- **F-5 — ArNS release-channel control rests on a [VERIFY].** The 3-of-5 controller model depends on unconfirmed ANT n-of-m capability; the fallback is a FROST-ed25519 threshold ceremony described in one bracket. If neither materializes, release-channel control degrades to single-key custody (§6.1 A7, FE-P7).

**Medium-high**

- **F-6 — No expedited release path** vs. emergency runtime upgrades (see X-7): 72h soak + attestations + 3-of-5 repoint while signing is disabled mid-epoch (§12.2, §18, §24.4).
- **F-7 — "Never transaction-critical" contradicts the doc's own definition.** §2.3 defines transaction-critical as anything changing what a user signs *"or their understanding of what signing does."* In a trading app, a poisoned provider-supplied price chart shapes trades; sampling is 1 row per 16 pages and self-consistent forgeries are admitted to be undetectable by sampling (§16.4). Pre-sign values are genuinely chain-verified — but the doc should either narrow its definition or declare chart-manipulation an accepted economic residual; currently it claims both.

**Medium (selected)**

- Boot state machine (§8.1) lacks failure transitions its own error taxonomy defines (worker spawn, WASM, IndexedDB, sync-phase degradation), and its relation to the §18 compat machine is undefined.
- The provider firewall is structural between packages but only a lint-pattern inside `apps/web`; provider-fed store data can seed transaction form state.
- Backfill math is internally inconsistent three ways (50 blk/s claimed ⇒ 12.5 days/hour, not "~9"; §21 budgets only 20 blk/s), and the §15.8 growth model appears to omit `Observed` rows (~15× underestimate of the events table).
- CSP `connect-src *` provides zero exfiltration containment; the stated compensating control addresses only code injection (§9.8).
- The required WSS bootnode set (~4 operators) is the real infrastructure floor of the "no required infrastructure" claim, with no threat-table row for total WSS loss/censorship (§11.9, §20).
- The historical-metadata fallback (ship blobs for every past spec_version) grows unboundedly against the bundle budget (§15.5 vs §21); multi-tab/SharedWorker behavior is a chain of FE-P3 unknowns with unbudgeted 2× smoldot memory on Android.

**Open questions:** 34 [VERIFY] tags mapped to prototypes FE-P1–FE-P9 — the honest but real message is that PAPI surface details, smoldot behavior (notably runtime-call verification, FE-P2 — pivotal), mobile budgets, and the entire ar.io operational layer are unproven. §26 has no team-size or calendar mapping, and FE-R1 sits on another team's roadmap.

## Concrete recommendations

Ordered; the first three block everything else.

1. **Freeze the integration contract in one owned place.** Lift frontend §13 (all 11 methods + view types + `RecentCohortSummaries`) plus the event schemas (`Traded`, `Observed`, and a defined oracle event set) into the backend's `futarchy-primitives`/`runtime-api` spec — or a shared `INTERFACE.md` — and either apply or formally reject each §30 patch item. While doing so, add the four items §30 itself missed: a WSS bootnode row in backend §4.2, oracle events, an ss58 prefix/paraId decision, and a `baseline_market` field on `EpochInfo` (fixes X-4, X-5, X-8).
2. **Rework the conditional ledger before freezing anything.** Extend `PositionKind`/vault structure to carry gate-market sets (7 kinds/branch for gated classes), define the Baseline market's collateral model (e.g., a per-epoch pseudo-vault), add a `Void` terminal state with its redemption calls, and restate the §10.2 conservation identity to cover all of it (B-1, B-2, B-4). Then re-confirm the frontend's client-recompute contract (C-5) against the new shapes.
3. **Specify the trade path.** State explicitly that `buy()` consumes branch-NUM (or auto-splits NUM and returns the mirror), and specify the LMSR revenue-splitting mechanic and `headroom` sizing that make the solvency claim true (B-7).
4. **Fix the numbers.** Correct vector V1 to 512.4948…; reconcile the market-count bounds (31/121/124/225) and cohort decompositions into one table; fix §30.1's maturity arithmetic and the IntakeQueue-vs-MaxLiveProposals contradiction (B-3, medium items).
5. **Close the two cheap griefing levers.** Scope the dispute-veto (e.g., only disputes opened before the decide window, or require surviving round 1 before it blocks decisions) and make repeated `NotDecisionGrade` outcomes burn a bond fraction so slot-crowding has a real price (B-10, slot-crowding). Bound or price account-keyed state growth (B-9).
6. **Decide who serves the values layer.** Either add a referenda/voting/ratification surface to the frontend's scope (it can be minimal: list, vote, delegate — needed for META ratification and oracle adjudication), or amend INV-FE-4's "every protocol workflow" claim and name the tool that fills the gap (X-2). Simultaneously redesign ratification timing so the referendum runs during the timelock — queue-time requires *ongoing*, execute-time requires *Passed* (B-8).
7. **Pick a deep-history owner and make the docs match.** Options: require bootnode/RPC operators to retain N days of state (an honest ops line in backend §4.2), promote snapshots to a soft-required first-visit path, or accept 8-checkpoint charts for provider-less users — then fix frontend §15.4/§15.6/E3 and U-3, and reword P-4 to match reality (X-3, F-1). Independently, fix the §11.8 promotion rule to "re-fetched through smoldot only" (F-2).
8. **Define an expedited-release procedure** for descriptor-only frontend updates (pre-authorized, skip soak, still 2 attestations), and quantify the acceptable read-only window during emergency upgrades against users' time-boxed obligations (X-7, F-6).
9. **Publish the normative inputs.** Commit the INV-FE-1…15 / question-block source document to the repo; fix the phantom §18.5/§18.6 references; define `DispatchOutcomeCode` and make the `RejectReason` variants reachable (F-3, backend enum drift).
10. **Extend the threat models where money is already real:** a Phase-3 insider/sudo-compromise row with a cap on real-NUM exposure while sudo lives; a USDC issuer/Asset Hub freeze analysis; a preimage-unnote rule (request/hold the preimage at Screening) (B-12, USDC freeze, preimage TOCTOU).

Worth preserving on both sides: the backend's origin discipline, bounded-hooks story, and status-quo-default rigor are genuinely strong; the frontend's `Verified<T>` provenance typing, dual-derivation tripwires (quote recompute as an integrity check), release-scoped service worker, and [VERIFY]-to-prototype discipline are the right patterns — the issues above are almost all specification gaps at the seams, not flaws in the core architectural bets.
